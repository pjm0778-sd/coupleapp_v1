## 1. Overview

### 1.1 Purpose

사용자가 안드로이드, 아이폰, 구글 캘린더, 기타 달력 앱 등 다양한 형태의 달력 이미지를 캡처해서 업로드했을 때, 막대형 다중 일정과 하루 내 중복 일정을 포함한 모든 일정을 빠짐없이 추출할 수 있도록 OCR Edge Function의 프롬프트를 재설계합니다.

### 1.2 Background

**현재 프롬프트의 핵심 한계**

| 한계 | 설명 |
|------|------|
| 다양한 달력 앱 대응 불가 | 특정 레이아웃을 가정한 분석 지침이 달력마다 다른 UI에서 오동작 |
| 막대형 다중 일정 미인식 | 구글 캘린더 등의 multi-day event bar를 날짜별로 파악하지 못함 |
| 하루 중복 일정 누락 | 같은 날 여러 일정이 쌓인 경우 일부만 추출하거나 누락 |
| 과도한 형식 제약 | 단계별 지침이 GPT-4o의 자유로운 이미지 해석을 오히려 방해 |

### 1.3 Scope

#### In Scope

- [ ] `useMapping=true` 프롬프트 재작성
- [ ] `useMapping=false` 프롬프트 재작성
- [ ] JSON 스키마 변경: `date` → `start_date` / `end_date`
- [ ] 막대형 다중 일정 인식 지원
- [ ] 하루 중복 일정 전체 추출 지원
- [ ] JSON 파싱 로직 보강 (코드블록 포함 응답 대응)

#### Out of Scope

- [ ] GPT-4o 외 다른 모델 전환
- [ ] Flutter 모델/UI 변경 (별도 작업)
- [ ] DB 스키마 변경 (별도 작업)

## 2. 프롬프트 설계 방향

### 2.1 핵심 원칙

**기존**: 단계별 번호 지침 → GPT-4o가 틀에 맞춰 분석
**신규**: 목표와 출력 형식만 명시 → GPT-4o가 이미지를 자유롭게 해석

GPT-4o는 이미지 이해 능력이 뛰어나므로, "무엇을 뽑아내야 하는지"와 "어떤 형식으로 줘야 하는지"만 명확히 주고, "어떻게 분석하라"는 지침은 최소화합니다.

### 2.2 공통으로 처리해야 할 이미지 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| 단일 일정 셀 | 날짜 칸에 색상 또는 텍스트 | 기본 근무표 |
| 하루 중복 일정 | 한 날짜에 일정 2개 이상 | 구글 캘린더 스택 |
| 막대형 다중 일정 | 여러 날에 걸친 가로 막대 | 구글/아이폰 캘린더 이벤트 바 |
| 시간 표시형 | 일정 옆에 시간 텍스트 포함 | "09:00 회의" |
| 색상 전용 | 텍스트 없이 배경색으로만 구분 | 근무표 앱 |

## 3. 새 JSON 스키마

### 3.1 변경 사항

| 필드 | 기존 | 변경 |
|------|------|------|
| `date` | "YYYY-MM-DD" | 제거 |
| `start_date` | 없음 | 추가 — 일정 시작일 |
| `end_date` | 없음 | 추가 — 일정 종료일 (하루짜리이면 start_date와 동일) |
| `work_type` | 유지 | 유지 |
| `color_hex` | 유지 | 유지 |
| `start_time` | 유지 | 유지 (없으면 생략) |
| `end_time` | 유지 | 유지 (없으면 생략) |

### 3.2 새 응답 예시

```json
{
  "year": 2026,
  "month": 3,
  "schedules": [
    {
      "start_date": "2026-03-10",
      "end_date": "2026-03-13",
      "work_type": "출장",
      "color_hex": "#4285F4"
    },
    {
      "start_date": "2026-03-10",
      "end_date": "2026-03-10",
      "work_type": "팀 회의",
      "color_hex": "#EA4335",
      "start_time": "14:00",
      "end_time": "15:00"
    },
    {
      "start_date": "2026-03-15",
      "end_date": "2026-03-15",
      "work_type": "야간",
      "color_hex": "#333333",
      "start_time": "22:00",
      "end_time": "06:00"
    }
  ]
}
```

## 4. 프롬프트 초안 방향

### 4.1 useMapping=false (자유 인식 모드)

- 역할: "달력 이미지에서 모든 일정을 추출하는 전문가"
- 지시: 이미지에 보이는 모든 일정(막대형 포함, 중복 포함)을 빠짐없이 추출
- 형식만 명시, 분석 방법 지침 최소화
- 연도/월 기본값만 제공

### 4.2 useMapping=true (매핑 참고 모드)

- useMapping=false의 추출 역할 + 매핑 테이블 참고
- 매핑 색상과 유사한 색상이면 해당 work_type 적용
- 매칭 안 되면 이미지에서 읽은 텍스트 또는 색상 그대로 사용

### 4.3 공통 제약

- JSON 코드블록(```) 없이 순수 JSON만 응답
- 해당 월의 날짜 범위를 벗어나는 일정 제외
- 일정이 없는 날짜는 포함하지 않음

## 5. JSON 파싱 로직 보강

### 5.1 현재

```typescript
const match = content.match(/\{[\s\S]*\}/)
```

### 5.2 변경

```typescript
const cleaned = content
  .replace(/```json\s*/gi, '')
  .replace(/```\s*/g, '')
  .trim()
const match = cleaned.match(/\{[\s\S]*\}/)
```

## 7. 성공 기준

| 기준 | 목표 |

## 8. Risks and Mitigation

| Risk | Mitigation |
|------|------------|
| 자유도를 높이면 응답 형식이 깨질 수 있음 | JSON 형식 지침은 명확하게 유지 |
| 막대형 일정의 정확한 종료일 오인식 | 프롬프트에 막대 끝 날짜 확인 명시 |
| 달력 앱마다 레이아웃이 너무 다름 | GPT-4o의 시각적 판단에 위임 (단계 지침 제거 이유) |
