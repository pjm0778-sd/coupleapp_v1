# Plan: midpoint-date (중간지점 데이트 추천)

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 장거리 커플이 만날 장소를 정할 때 "어디서 만날까?"를 각자 따로 검색하며 시간을 낭비하고, 한 쪽이 더 오래 이동하는 불공평한 상황이 반복된다 |
| **Solution** | 두 사람의 출발지·교통수단(대중교통/자차)을 입력하면 이동시간 기준 공평한 중간지점 도시를 AI가 추론하고, 자동차 경로/비용·대중교통(지하철·KTX·고속버스) 정보·주변 맛집·데이트 코스를 한 화면에 제공한다 |
| **Function UX Effect** | 출발지 → 대중교통/자차 2택 선택(자차는 일반/전기 구분) → 테마 선택 → 중간지점 도시 + 각자 경로 비교표 + 주변 장소 추천을 한 번에 확인, 마음에 드는 장소를 캘린더 일정으로 바로 추가 가능 |
| **Core Value** | "어디서 만나지?" 고민을 앱 하나로 해결 — 공평한 중간지점 + 데이트 코스까지 원스톱 제공 |

---

## 1. 배경 및 목적

### 1.1 문제 정의
- 장거리 커플(서울-부산, 서울-대전 등)은 만남 장소 선정 시 각자 지도 앱을 따로 열어 검색
- 단순 지리적 중간점은 교통 접근성이 한 쪽에 불리할 수 있음 (지리적 중간 ≠ 이동시간 중간)
- 경로·비용·장소 추천을 한 앱에서 통합 확인하는 서비스 부재

### 1.2 목표
- **Phase 1 (MVP)**: 동일 교통수단 기준, AI 중간지점 추론 + 경로/비용 비교 + 장소 추천
- **Phase 2**: 혼합 교통수단, 실시간 유가, 전기차 충전소 정보

### 1.3 타겟 유저
- 주 사용층: 서로 다른 도시에 사는 장거리 연애 커플
- 보조 사용층: 같은 도시지만 동네가 멀어 중간지점이 필요한 커플

## 2. 기능 요구사항

### 2.1 입력 단계

- [ ] FR-01: 내 출발지 입력 (텍스트 검색 → Naver Geocoding으로 좌표 변환)
- [ ] FR-02: 상대방 출발지 입력 (동일 방식)
- [ ] FR-03: 교통수단 선택 — 각자 독립 선택, **2가지 옵션만 제공**
  - **대중교통**: 시스템이 출발지·도착지 거리에 따라 지하철·KTX/SRT·고속버스 중 최적 수단 자동 결정 (ODsay 연동)
    - 동일 생활권(약 50km 이내): 지하철/도시철도 우선
    - 광역(50~200km): 고속버스 또는 KTX/SRT
    - 장거리(200km 이상): KTX/SRT 우선
  - **자차**: 일반차 또는 전기차 구분 선택 (FR-04)
- [ ] FR-04: 자차 선택 시 차량 종류 선택 (비용 계산 기준 분기)
  - 일반차: 거리(km) ÷ 12(km/L) × 1,700원/L + 고속도로 통행료
  - 전기차: 거리(km) ÷ 6(km/kWh) × 300원/kWh + 고속도로 통행료
- [ ] FR-05: 테마 선택
  - 데이트 (맛집 + 카페 + 감성 스팟 중심)
  - 여행 (관광지 + 숙박 + 체험 중심)
  - 단순 중간지점 (거리 최소화 우선)

### 2.2 중간지점 추론 (Claude API)

- [ ] FR-06: 두 출발지 + 각자 교통수단 + 테마를 Claude API에 전달
- [ ] FR-07: Claude가 추천 중간지점 도시 2~3곳 반환
  - 이동시간 동등성 고려 ("두 사람이 비슷한 시간을 이동하는 지점")
  - 테마에 맞는 도시 특성 반영 (데이트 → 카페 많은 도시, 여행 → 관광지 있는 도시)
- [ ] FR-08: 추천 이유를 자연어로 생성 (왜 이 도시인지 설명)

### 2.3 경로 및 비용 계산

**자차 경로 (Naver Directions 5 API)**
- [ ] FR-09: 출발지 → 중간지점까지 자동차 경로 조회 (거리, 예상 소요시간)
- [ ] FR-10: FR-04의 차량 종류 기준 연비 비용 계산 (상세는 FR-04 참고)
  - 고속도로 통행료: 거리(km) × 50원 (추정값)

**대중교통 (ODsay — 이미 앱에 있음)**
- [ ] FR-11: 출발지·중간지점 거리 기반 최적 대중교통 수단 자동 결정
  - 단거리(~50km): 지하철/도시철도 경로 조회 (ODsay searchPubTransPathT)
  - 중거리(50~200km): 고속버스 시간표 + 요금 조회 (ODsay searchInterBusSchedule)
  - 장거리(200km+): KTX/SRT 출발역 자동 매핑 + 열차 시간표·요금 조회 (ODsay trainServiceTime)
- [ ] FR-12: 경로 결과에 수단 종류 명시 (지하철 / 버스 / KTX 등 UI 표시)

### 2.4 장소 추천 (Kakao Local API)

- [ ] FR-14: 중간지점 도시 좌표 기준 반경 내 장소 검색
- [ ] FR-15: 테마별 카테고리 필터링
  - 데이트: 음식점(FD6) + 카페(CE7)
  - 여행: 관광명소(AT4) + 숙박(AD5)
- [ ] FR-16: 장소 목록 표시 (이름, 카테고리, 거리, 카카오맵 링크)
- [ ] FR-17: 장소 선택 → 앱 내 캘린더 일정 추가 연동

### 2.5 결과 화면

- [ ] FR-18: 중간지점 도시 카드 (2~3곳, 가로 스크롤)
- [ ] FR-19: 도시 선택 시 양측 경로 비교표 표시

| 구분 | 나 | 상대방 |
|------|-----|-------|
| 교통수단 | 대중교통 (KTX) | 자차 (일반차) |
| 출발지 | 서울역 | 부산 해운대구 |
| 거리 | 약 250km | 약 120km |
| 소요시간 | 약 1시간 20분 | 약 1시간 30분 |
| 비용 | 약 47,000원 | 약 17,000원 |

- [ ] FR-20: 추천 이유 텍스트 (Claude 생성)
- [ ] FR-21: 주변 장소 목록 (최대 10곳)
- [ ] FR-22: Naver Dynamic Map에 중간지점 + 주변 장소 핀 표시

## 3. 기술 스택 및 API

| 역할 | API | 무료 제공량 | 비고 |
|------|-----|------------|------|
| 자동차 경로 | Naver Directions 5 | 6만 건/월 | 경유지 5개까지 |
| 지도 표시 | Naver Dynamic Map | 600만 건/월 | Android/iOS SDK |
| 주소 → 좌표 | Naver Geocoding | 300만 건/월 | |
| 대중교통 경로 | ODsay (기존) | 기존 연동 | 추가 비용 없음 |
| 장소 검색 | Kakao Local API | 무료 | 기존 앱 등록 |
| 중간지점 추론 | Claude API | 사용량 기반 | 검색당 1 call |

### API 사용량 추정 (월 활성 유저 500명 기준)
- 1인당 월 평균 3회 검색
- 검색 1회 = Directions 6회 + Geocoding 2회 + Kakao Local 3회 + Claude 1회
- Directions: 500 × 3 × 6 = **9,000건** (무료 6만 건 대비 15% 사용)
- **비용: 실질적으로 무료 범위 내**

## 4. 비기능 요구사항

- **응답시간**: 전체 결과 로딩 5초 이내 (Claude + Naver + Kakao 병렬 호출)
- **에러 처리**: API 실패 시 graceful degradation (일부 결과만 표시)
- **캐싱**: 동일 출발지 조합 결과 30분 캐시 (Naver Directions 호출 절감)
- **오프라인**: 네트워크 없을 시 안내 메시지 표시

---

## 5. 구현 범위 (Phase 1 MVP)

### In Scope
- 단일 교통수단 (양측 동일 수단 선택)
- 고정 연비/유가 기준 비용 계산 (실시간 유가 X)
- 중간지점 도시 2~3곳 추천
- 주변 맛집/카페/관광지 최대 10곳

### Out of Scope (Phase 2)
- 혼합 교통수단 (A는 KTX, B는 자차)
- 실시간 유가 반영 (Opinet API)
- 전기차 충전소 위치 표시
- 커플 간 장소 투표 기능
